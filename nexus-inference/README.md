# nexus-inference

Real-time CPU inference for small, pre-trained models.

This is not a GPU framework. There is no training, no automatic
differentiation, no batch processing. nexus-inference runs tiny models
— the kind that sit in an event loop and make decisions at wire speed.
Sub-microsecond prediction on the hot path, zero allocation after
construction.

**Design point:** Models are trained in external frameworks (LightGBM,
PyTorch), exported as flat weight arrays, loaded once via `from_parts`,
and then called millions of times. Every type pre-allocates all scratch
buffers at construction. The inference methods (`predict`, `step`) touch
only stack and pre-allocated memory.

## Model Types

### Gradient-Boosted Decision Trees — `GbdtF64` / `GbdtF32`

Ensemble of decision trees. Each tree partitions features by threshold
comparisons, traversing to a leaf value. The final prediction is the sum
of all leaf values plus a base score.

16-byte nodes in false-branch-next (depth-first) layout — the right
child is always `idx + 1`, so ~50% of traversals are sequential memory
access served by the hardware prefetcher. NaN-aware routing matches
LightGBM semantics. Partial ensemble evaluation via `predict_n`.

**Best for:** Tabular features with clear names and semantics —
microstructure metrics (book imbalance, spread, queue depth), windowed
statistics (VPIN, volatility ratios), categorical flags. GBDTs handle
feature interactions, missing values, and heterogeneous scales natively.
They don't need feature normalization and are interpretable through
feature importance.

**Guidance:** Use `GbdtF64` when loading LightGBM models (LightGBM
trains in f64 natively). `GbdtF32` is available when memory or
throughput matters more than precision. For temporal patterns, encode
time through multi-timescale windowed features (e.g., VPIN at 1s, 10s,
60s windows) rather than expecting the tree to learn temporal
dependencies — GBDTs see each sample independently.

### Multi-Layer Perceptron — `MlpF64` / `MlpF32`

Feedforward neural network. Dense layers connected by nonlinear
activations (ReLU, Tanh, Sigmoid, GELU, Swish, ELU, LeakyReLU, or
Identity). Weights are row-major matching PyTorch's `nn.Linear` layout.
SIMD-accelerated dot products (4-at-a-time tiling with AVX2/FMA
dispatch).

**Best for:** Combining pre-computed signals into a single score.
The classic use case is a two-stage pipeline: individual signals are
generated by GBDTs, streaming statistics, or other feature extractors,
then stacked into a dense vector and fed to a small MLP that learns
nonlinear combinations. Also useful as a lightweight classifier or
regressor over dense, normalized feature vectors (embeddings, PCA
outputs, learned representations).

**Guidance:** Keep hidden layers small — `[n_signals, 16, 1]` or
`[n_signals, 32, 16, 1]` is typical. Larger architectures belong on a
GPU. The output layer is always linear; apply sigmoid or softmax in
your own code if needed for classification. Use `MlpF32` for
PyTorch-trained models (PyTorch defaults to f32). `MlpF64` is available
for precision-sensitive applications.

### Lookup Table — `LutF64` / `LutF32`

Pre-computed prediction table indexed by discretized features. Each
feature is mapped to a bin via uniform spacing, and the bin indices
address a flat table. O(1) prediction — one division and one array
lookup per feature, no arithmetic beyond indexing.

**Best for:** Extremely latency-sensitive paths where even a tree
traversal is too much. Effective when features are low-dimensional
(1-3 features) and the relationship can be captured at the bin
resolution. Pre-compute the table offline from any model.

**Guidance:** Table size grows exponentially with feature count
(`n_bins ^ n_features`). Practical for 1-3 features at moderate bin
counts. Out-of-range features are clamped; NaN maps to bin 0.

### LSTM — `TinyLstmF32`

Single-layer Long Short-Term Memory network. Four gates (input, forget,
cell candidate, output) with hidden and cell state carried between
`step` calls. Weight parameters map directly to PyTorch's `nn.LSTM`
tensors — gate order is input, forget, cell candidate, output.

Internally fuses `weight_ih` and `weight_hh` into a single
`(4H, I+H)` matrix at construction for one matmul per step instead
of two. Biases pre-summed. Same SIMD dot product path as MLP.

**Best for:** Streaming sequence modeling where the model needs to
remember and selectively forget over time. The cell state acts as
long-term memory gated by learned sigmoid gates — good for patterns
like accumulating flow toxicity over a session, tracking regime
persistence, or detecting gradual shifts that unfold over many
timesteps. The forget gate lets the model decide when past state is
no longer relevant.

**Guidance:** Hidden sizes of 8-64 are the target range for real-time
use. Larger models should run on GPU. LSTM is more expressive than GRU
(separate cell state + 4 gates) but costs ~33% more compute. If your
temporal patterns are simple and you need every nanosecond, consider
GRU. Gate activations are hardcoded (sigmoid/tanh) — this is the
standard formulation, not configurable.

### GRU — `TinyGruF32`

Single-layer Gated Recurrent Unit. Three gates (reset, update,
candidate) with hidden state carried between `step` calls. ~75% of
LSTM compute — three gates instead of four, no separate cell state.
Weight parameters map directly to PyTorch's `nn.GRU` tensors. Uses
PyTorch's default formulation where reset is applied after the
hidden-to-hidden matmul.

**Best for:** Same temporal modeling as LSTM but with a simpler memory
model. On many tasks GRU matches LSTM quality with less compute. Good
default when you don't have a reason to prefer LSTM — try both and
keep whichever trains better.

**Guidance:** Same hidden size range as LSTM (8-64). GRU uses two
matmuls per step instead of LSTM's one fused matmul (required because
the candidate gate applies the reset gate between the input-to-hidden
and hidden-to-hidden products). Despite the extra matmul call, total
FMA count is lower. Gate activations are hardcoded (sigmoid/tanh).

### Causal 1D Convolution — `Causal1dConvF32`

Streaming causal convolution over a sliding window. Maintains a
circular buffer of the last `kernel_size` inputs. Each step writes
the new input, convolves the buffer with learned filters, applies a
configurable activation, and projects to output. Causal: only past
and current inputs contribute — no future leakage.

**Best for:** Detecting short-range temporal patterns with a fixed
receptive field. Unlike LSTM/GRU, the convolution has a hard horizon
(the kernel width) — it sees exactly the last K timesteps, no more.
Good for local pattern detection: short-term momentum, micro-bursts
in order flow, periodic signals with known period. Cheaper than
recurrent models when you don't need long-range memory.

**Guidance:** The `is_primed()` method returns `false` until
`kernel_size` steps have been processed — output before priming uses
a partially filled buffer (zero-padded). Activation is configurable
(same set as MLP) because the convolution is a feature extractor,
not a gating mechanism. Small kernels (3-8) and moderate filter counts
(8-32) are typical for real-time use.

## Choosing a Model

| Scenario | Model | Why |
|----------|-------|-----|
| Tabular features, named metrics | GBDT | Native feature interactions, no normalization needed |
| Combining multiple signals | MLP | Learns nonlinear combinations of pre-computed features |
| Ultra-low-latency, few features | LUT | O(1), pre-computed, no arithmetic |
| Temporal patterns, long memory | LSTM | Cell state retains information over many steps |
| Temporal patterns, fast & simple | GRU | Simpler than LSTM, often comparable quality |
| Short-range temporal, fixed window | Conv | Fixed receptive field, cheaper than recurrent |
| Two-stage pipeline | GBDT → MLP | Signals from trees, combined by network |
| Streaming regime detection | LSTM or GRU | Hidden state accumulates evidence over time |

## Features

- `std` (default) — standard library support
- `alloc` — enables MLP, LUT, GBDT, Conv types (heap allocation for weight storage)
- `libm` — no_std math fallback (enables LSTM/GRU without std)
- `loader-lightgbm` — LightGBM text format parser (implies `std`)

## Usage

```rust
use nexus_inference::{GbdtF64, MlpF32, TinyLstmF32, Activation};

// GBDT: load and predict
let model = GbdtF64::from_lightgbm(model_bytes).unwrap();
let score = model.predict(&features);

// MLP: dense signal combination
let mut mlp = MlpF32::from_parts(
    &[4, 16, 1], &weights, &biases, Activation::Relu,
).unwrap();
let signal = mlp.predict(&feature_vector);

// LSTM: streaming temporal inference
let mut lstm = TinyLstmF32::from_parts(
    4, 8, 1,
    &weight_ih, &weight_hh, &bias_ih, &bias_hh,
    &w_out, &b_out,
).unwrap();
let score = lstm.step(&input_frame);     // first timestep
let score = lstm.step(&next_frame);      // carries hidden state forward
lstm.reset_state();                      // clear for new sequence
```

## Performance

Target latency for typical configurations (single-threaded, AVX2+FMA):

| Type | Configuration | Target |
|------|---------------|--------|
| GBDT | 50 trees × depth 6, 8 features | < 500 ns |
| GBDT | 100 trees × depth 6, 8 features | < 1 µs |
| MLP  | 8 → 16 → 1, ReLU | < 100 ns |
| LSTM | hidden=16, input=8 | ~250 ns |
| GRU  | hidden=16, input=8 | ~200 ns |
| Conv | 8 filters, kernel=4, 4 channels | ~50 ns |
